Alembic baseline consolidation — plan and deploy runbook

GOAL
  Replace 26 accumulated migration files (with duplicate revision IDs and
  orphan heads) with a single baseline migration that creates the entire
  schema from the current SQLAlchemy models. Acceptable because we are
  wiping the devbox DB as part of this deploy.

CODE CHANGES
  1. Delete every file in backend/alembic/versions/*.py
  2. Add one new baseline migration that:
     a. Creates the pgvector extension (CREATE EXTENSION IF NOT EXISTS vector)
     b. Calls Base.metadata.create_all on the bound connection to emit
        CREATE TABLE for every model declared in news_service.models
     c. Uses down_revision = None (it is the first migration)
  3. Leave alembic.ini, alembic/env.py, alembic/README, script.py.mako
     untouched — those are the harness, not history
  4. Leave the alembic dependency in pyproject.toml — env.py still uses it

DEPLOY RUNBOOK (devbox)
  1. Stop the stack:
       docker --context devbox compose down
  2. Drop the DB volume so postgres starts empty:
       docker --context devbox volume rm news-monorepo_postgres_data
       (or whatever the volume name is — inspect with `docker volume ls`)
  3. Bring everything back up:
       docker --context devbox compose up --build -d
  4. Backend container runs `alembic upgrade head` at startup, which applies
     the single baseline against the empty DB → full schema in one step,
     alembic_version row stamped with the baseline revision ID
  5. Verify:
       docker --context devbox compose logs backend | tail -n 40
       docker --context devbox compose exec postgres psql -U news -d news \
         -c '\dt' -c 'SELECT version_num FROM alembic_version;'

NOT DOING
  - Not keeping any historical migrations — they are all superseded by the
    baseline and the data they used to preserve is being dropped anyway
  - Not dropping alembic entirely (path A). Keeping the harness lets us
    evolve schema normally from this baseline forward
